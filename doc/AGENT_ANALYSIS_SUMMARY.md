# 三面向專家分析總結 (Three-Agent Analysis Summary)

**日期**：2026-02-28
**目的**：從 UX、技術、風險管理三個角度分析 Gem3 轉型計劃

---

## 📊 執行摘要 (Executive Summary)

### 結論
✅ **Gem3 B2B 轉型在技術上是可行的** (可行性評分：8/10)，但**複雜性比預期高**，特別是在以下方面：
1. 認證層整合 (NextAuth + PocketBase)
2. 月結系統實現 (當前無此系統)
3. 零停機資料庫切換

### 建議
- **順序**：從低風險路由開始逐步遷移 (已有 6 個 PocketBase 路由作為概念證明)
- **時程**：13-20 週 (取決於團隊規模和並行度)
- **人力**：2-3 人開發團隊
- **關鍵成功因素**：前期與業務確認需求，特別是月結流程

## 🎨 Agent 1: UX/產品分析

### 當前 B2C 複雜性
CEO 平台為**成熟的團購平台**，包含：

**前端複雜功能** (30-40% 代碼可移除)
- 首頁：搜尋欄、行銷區塊、特色/最新產品輪播
- 產品列表：全文搜尋、多層分類、4 個排序選項、分頁
- 產品詳情：
  - 分層定價引擎 (基於數量自動計算)
  - 團購進度條 (實時追蹤目標數量)
  - 倒計時計時器 (時間限制)
  - 5 星評分 + 銷售計數
  - "下一層級折扣" 誘導提示

**管理後台複雜度** (20-30% 代碼可移除)
- 8 部分導航 + 9 個儀表板指標
- 進階分析：訂單狀態分佈、營收趨勢、熱門產品、最近訊息
- 進階過濾：成員搜尋、狀態篩選、多列排序

### B2B 簡化機會 (Quick Wins)

| 項目 | 移除 | 預期影響 |
|------|------|--------|
| 首頁搜尋 | ✂️ | -40% 代碼 |
| 分層分類樹 | ✂️ | 簡化為下拉式 |
| 全文搜尋 | ✂️ | 移除搜尋 API |
| 分層定價網格 | ✂️ | 改為單價或 2 層 |
| 進度條/倒計時 | ✂️ | 刪除 UI 元件 |
| 評分系統 | ✂️ | 移除或靜態 5 星 |
| 儀表板圖表 | ✂️ | 保留 3 張卡片 |

### 保留的核心 B2B 功能
- ✅ 基本產品瀏覽與過濾
- ✅ 購物車管理
- ✅ 結帳 (3 種支付方式)
- ✅ 訂單歷史
- ✅ 產品管理 (CRUD)
- ✅ 訂單管理
- ✅ 使用者管理
- ✅ FAQ/說明文檔

### 預期成果
- **代碼減少**：30-40% 前端 + 20-30% 管理元件
- **效能提升**：更快加載、更少複雜 JavaScript
- **可維護性提升**：更簡單的元件樹、更少的狀態管理

## 🏗️ Agent 2: 技術架構分析

### 當前混合狀態 (Hybrid Database Status)
```
41 Prisma 路由     ╭─ PostgreSQL ──╮
(Admin, Auth)  ───┤               ├─ ?
6 PocketBase 路由  ╰─ PocketBase ──╯
(Cart, Orders)
```

**問題**：
- Prisma 是**幽靈依賴** (`package.json` 有命令但沒有依賴宣告)
- NextAuth + JWT 與 PocketBase Auth 系統不匹配
- 41 個路由使用 Prisma；6 個已使用 PocketBase (概念證明)

### PocketBase 遷移差距分析

**必需集合** (最小 Schema)
```
users          → 40 字段 (email, taxId, name, points, etc)
products       → 13 字段 (name, price, category, firm, etc)
categories     → 6 字段 (id, name, parentId, sortOrder, isActive)
orders         → 9 字段 (orderNo, userId, status, totalAmount, etc)
order_items    → 5 字段 (orderId, productId, quantity, price)
cart_items     → 3 字段 (userId, productId, quantity)
point_txns     → 5 字段 (userId, type, amount, reason)
```

### 關鍵複雜性

| 功能 | PostgreSQL/Prisma | PocketBase | 遷移難度 |
|------|------------------|-----------|--------|
| **事務** | `prisma.$transaction()` | 需要手動編排 | 🔴 高 |
| **複雜查詢** | `include()` 巢狀 | 多次 API 呼叫 | 🟡 中 |
| **聚合** | `count()`, `sum()` | 手動計算 | 🟡 中 |
| **FK 約束** | 資料庫強制 | 應用層驗證 | 🟡 中 |
| **全文搜尋** | PostgreSQL FTS | 字串比對 | 🟡 中 |
| **批量操作** | `createMany()` | 迴圈 + API 呼叫 | 🟡 中 |
| **認證** | NextAuth + Prisma | PocketBase Auth? | 🔴 高 |

### 高風險遷移路由

**第 4 波 (最複雜)**
- `/api/admin/products/[id]` - 產品 + 價格層級事務
- `/api/admin/orders/[id]` - 訂單 + 項目 + 支付關係
- `/api/admin/users/[id]` - 用戶 + 點數交易 + 日誌

**第 3 波 (認證相關)**
- `/api/auth/login` - Prisma 用戶查詢
- `/api/auth/register` - OAuth 同步
- `/api/auth/oauth/*` - OAuth 邏輯

### 關鍵檔案 (必須優先處理)
1. **`/src/auth.ts`** - 認證核心 (所有 41 個路由依賴)
2. **`/src/lib/auth-helper.ts`** - Bearer Token 驗證 (3 個 Prisma 查詢)
3. **`/prisma/schema.prisma`** - 20+ 模型定義
4. **`/api/admin/products/route.ts`** - 最複雜的事務
5. **`/api/admin/users/route.ts`** - 複雜查詢代表

### 遷移時程估計

| 波次 | 路由數 | 難度 | 人週 |
|------|--------|------|------|
| 1️⃣ (已有 PocketBase) | 6 | 低 | 1 |
| 2️⃣ (公開路由) | 6 | 低 | 2 |
| 3️⃣ (認證) | 8 | 高 | 4 |
| 4️⃣ (管理) | 23 | 高 | 10 |
| **總計** | **43** | | **17** |

**+3-4 週準備、測試、上線** → **6-7 週總計**

## ⚠️ Agent 3: 風險與反對意見分析

### 高風險 (🔴 需立即解決)

#### 1. 認證層複雜性
**現狀**：
- Desktop: NextAuth session → API endpoints
- Mobile: Bearer JWT → API endpoints (via auth-helper)

**問題**：
- PocketBase 有自己的認證系統
- 整合兩者會造成架構分裂
- 2FA (EmailVerification token) 可能與 PocketBase 不匹配

**成本**：所有 41 個 API 路由需要重寫認證層

#### 2. 月結系統 (月結不存在於當前代碼！)
**假設**："簡化支付為 cash/monthly/points"

**現實**：
- 當前 Order model 無 `billingPeriod` 或 `dueDate` 字段
- PointTransaction 無帳戶額度限制
- 無月度發票生成邏輯
- 無逾期提醒工作流

**需要實現**：
- 月度發票生成排程器
- 應收帳款追蹤
- 點數兌換率定義
- 稅務合規 (臺灣電子發票)

**成本**：額外 2-3 週開發 + 與業務確認需求

#### 3. 零停機切換
**假設**："無停機時間切換"

**現實**：
- PostgreSQL → PocketBase 需要資料形狀轉換
- 40+ 資料表映射至 PocketBase 集合
- 活躍訂單在過渡期間可能遺失
- 認證令牌在切換時會失效

**成本**：需要並行運行、完整資料驗證、回滾計劃

### 中等風險 (🟡 需監控)

1. **PocketBase 效能**
   - 在大量並發時可能達到限制
   - 無原生 N+1 查詢解決方案

2. **複雜查詢遷移**
   - Prisma 嵌套 include() 在 PocketBase 中難以實現
   - 某些報表/分析查詢需要重新設計

3. **資料完整性**
   - 無外鍵約束在 PocketBase 中
   - 需要應用層驗證

### 低風險 (🟢 可控)

1. 代碼清理 (易於復原)
2. UX 簡化 (可快速迭代修改)

### 未知未知 (Unknown Unknowns)

必須與業務確認：
- [ ] 現有多少用戶有**未完成的月結**？
- [ ] 當前使用**哪些支付方式**？
- [ ] **員工數量與交易量**？
- [ ] **搜尋功能使用率**？
- [ ] **合約中承諾的功能**？

## ✅ 推薦方案 (Recommended Approach)

### 遵循順序
1. **準備** (第 1 週-2 週)
   - 與業務確認月結、點數、支付詳情
   - 解決 Prisma 幽靈依賴
   - 設計 PocketBase Schema

2. **低風險波次優先** (第 3-6 週)
   - 遷移已使用 PocketBase 的 6 個路由 (概念驗證)
   - 遷移公開路由 (products, categories)
   - 充分測試、有信心後再進行

3. **認證層最後** (第 7-8 週)
   - 最複雜、影響最廣
   - 確保前面的波次已穩定

4. **管理路由並行** (第 6-11 週)
   - 與認證層並行
   - 優先處理 `admin/products` 和 `admin/orders`

5. **業務需求實現** (第 12-15 週)
   - 月結系統、點數系統
   - 發票生成、稅務合規

### 並行工作
- **UX 簡化可與資料庫遷移並行**
  - UX 簡化不依賴 PocketBase (可在 Prisma 上進行)
  - 先移除 UI，然後簡化 API

- **支付重構可延遲**
  - 不是第 1 優先，可在資料庫穩定後實施
